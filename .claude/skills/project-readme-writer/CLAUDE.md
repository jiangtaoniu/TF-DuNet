# 项目 README 与项目说明写作技能

## 目标

当用户要求编写 README、项目说明、简历项目介绍、面试项目介绍、GitHub/Gitee 项目主页文档时，必须把项目写得专业、清晰、可信、工程化。

文档要让 HR、面试官和专业开发者都能快速看懂项目价值，但不能写成浮夸宣传稿，也不能像随手做的 Demo。

## 核心原则

必须先分析真实项目，再写文档。

如果项目中有代码、目录、配置文件、接口文件、数据库脚本、Docker 配置、启动命令或页面文件，必须先查看这些内容，再总结项目。

禁止凭空编造功能、技术栈、架构、接口、部署方式、性能指标、用户规模、线上运行情况。

如果某项内容不能从代码或文件中确认，必须写成“可扩展方向”或“后续计划”，不能写成已经完成。

## 写作风格

使用中文为主，语言专业、务实、清楚。

避免使用以下空泛表达：
- 颠覆式
- 革命性
- 企业级闭环
- 极致性能
- 全链路智能化
- 高并发支撑海量用户
- 完美解决
- 行业领先

除非项目代码和部署证据能支撑，否则不要使用这些词。

推荐使用以下表达：
- 面向真实业务流程设计
- 具备清晰的角色权限边界
- 前后端职责划分明确
- 支持本地部署与二次开发
- 代码结构便于维护和扩展
- 围绕业务流程完成核心功能闭环
- 当前版本聚焦于核心业务验证和工程实现

## 必须先检查的项目内容

写 README 前，优先查看：
1. 项目根目录结构
2. `README.md`
3. `package.json` 或 `requirements.txt`
4. `pom.xml` 或 `CMakeLists.txt`
5. `docker-compose.yml`
6. `.env` 或配置文件
7. 前端路由或后端入口文件
8. API 封装目录或 Controller 层
9. 核心 Service 或核心算法（如 `C++` 节点）
10. 数据库初始化脚本或预处理脚本
11. 启动、构建、部署相关脚本（`*.sh`，`*.bat`）

根据真实文件判断项目类型、技术栈、启动方式和功能范围。

## README 推荐结构

优先按下面结构编写，根据项目实际情况灵活调整：

1. 项目名称：一句话说明项目核心定位。
2. 项目定位：解决什么业务/技术问题、面向什么场景。
3. 核心功能：按业务角色或技术模块介绍功能。
4. 技术栈：只写项目中真实存在的技术。
5. 系统架构或模块划分：系统间数据流转与协作关系。
6. 目录结构：关键目录作用说明。
7. 本地运行：从真实脚本中提取的可执行启动命令。
8. 核心业务流程：核心执行链路串讲。
9. 项目亮点：基于真实代码的工程设计（如内存预分配、请求封装等）。
10. 面试介绍篇（可选）：本人职责、技术难点、解决方案。
11. 后续优化方向：合理的扩展规划。

## 面试项目说明要求

用于简历/面试时重点阐述：
- 为什么要做这个项目（解决什么具体痛点）
- 你负责了哪些模块
- 项目里有哪些真实脱离 Demo 级别的工程架构设计（如资源解耦、防并发等）
- 把枯燥的“增删改查”或“加载模型”翻译背后的业务流、权限隔离或底层硬件调度原理讲清楚。

## 项目亮点写法要求

绝对具体，严禁虚化。

错误写法：本项目采用先进架构，实现高性能、高可用平台。
推荐写法：本项目基于 C++ 与 TensorRT 重构前向推理逻辑，预先锁定 40000 规模显存池，并采用贪心 NMS 在 CPU 兜底，将点云检测整体延迟控制在 7ms 内。

## 启动说明要求

必须核验脚本！
不能凭主观经验捏造常规范式启动。如果项目是用 `source install/setup.bash && ros2 launch ...` 启动，绝对不能写普通的 `cmake .. && make`。若需要更改环境变量、模型路径映射，必须显式说明。

## 输出前自检

1. 技术栈、启动流程、架构图信息是否全部来自真实文件？
2. 是否存在编造并发、QPS 数据、以及企业上线假象？
3. 对于待完善点，是否已老实标注为“未来优化”，而非宣称上线？
4. HR 看了懂业务定位，Dev 看了懂代码流，Tech Lead 看了见工程内功。

（核验完毕后，在文档末尾以开发者注释的方式表明本次解析已通过代码硬核自检。）
