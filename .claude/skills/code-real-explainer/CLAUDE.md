# 真实代码解释技能

## 核心原则

只要项目中存在对应代码，就必须基于真实代码解释，不允许用框架经验、常见原因或猜测代替代码分析。

如果已经找到代码，回答中不要使用“可能”“大概”“应该是”“估计”“看起来像”等不确定表达。  
只有在没有找到对应代码、缺少调用方、缺少配置文件或缺少日志上下文时，才可以说明“不确定”，并明确指出缺少什么证据。

## 必须执行的流程

1. 先定位代码  
   根据用户给出的文件名、页面名、函数名、接口路径、报错行号、日志关键字或路由地址，在项目中查找真实代码。

2. 再分析代码  
   解释代码的实际执行流程，包括入口位置、调用链、关键变量、条件分支、接口请求、返回处理、异常处理和最终结果。

3. 最后反向核验  
   在回答前必须重新检查一遍：  
   - 结论是否能在代码中找到依据；  
   - 文件名、函数名、接口路径是否真实存在；  
   - 执行顺序是否和代码一致；  
   - 是否把未确认内容说成了确定结论；  
   - 是否残留“可能、大概、应该是”等猜测性表达。

## 回答格式

回答代码问题时，按下面顺序输出：

1. 直接结论  
   先说明代码中已经确认的真实原因或真实逻辑。

2. 代码依据  
   说明依据来自哪个文件、哪个函数、哪个接口、哪个生命周期或哪个关键语句。

3. 执行流程  
   按真实代码顺序解释从入口到结果的过程。

4. 二次核验结果  
   明确说明：  
   “我已按当前代码反向核验，上述结论与已查看代码一致。”  
   如果有无法确认的部分，必须写成：  
   “当前代码中尚不能确认的是……，还需要查看……”

## 禁止行为

禁止在有代码的情况下直接说：

- 可能是接口问题
- 大概是权限问题
- 应该是路径不对
- 估计是后端没启动
- 看起来像 token 问题

应该改成：

- 当前代码中，报错来自……
- 真实触发位置是……
- 该接口实际调用的是……
- 当前执行流程确认是……
- 这部分代码没有进入……分支

## 特别要求

如果用户给的是前端页面问题，要追踪：

页面文件 -> 生命周期/点击事件 -> API 封装 -> request 工具 -> 后端接口路径。

如果用户给的是后端接口问题，要追踪：

Controller -> Service -> Mapper -> XML/SQL -> 数据库字段或返回对象。

如果用户给的是报错日志，要先定位报错文件和行号，再解释为什么这行代码会触发该日志或异常。

永远不要编造不存在的文件、函数、接口、字段或行号。
